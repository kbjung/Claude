> 펩타이드 프로젝트 전체 컨텍스트는 상위 폴더의 `CLAUDE.md`를 함께 참조한다.

## ⚠️ 시작 전 필독: 양식 보존을 위한 핵심 원칙

다른 세션에서 작업할 때 **반드시 이 원칙을 지켜야 양식이 깨지지 않는다**. 과거 v2.4에서 새로 처음부터 작성한 결과 730단락(v2.3의 절반)으로 양식과 깊이가 달라진 사례가 있다.

### 절대 원칙

1. **새로 처음부터 generate_v*.js 작성 금지**. 반드시 가장 최신 generate_v*.js (현재 `generate_v7.js`)를 `cp`로 복사한 후 **Edit 도구로 부분 수정**만 한다.
2. **헬퍼 함수(h1, h2, h3, p, bullet, easyBox, quoteBox, makeTable, modelSubHead)만 사용**. raw `new Paragraph()` 또는 `new Table()` 직접 작성 금지.
3. **6단 구조(A-F) 유지**. 모델 챕터는 반드시 A(한눈에) → B(쉬운 설명) → C(생명공학 배경) → D(동작 로직) → E(검증 근거) → F(한계와 적용 타당성).
4. **색상·여백·다이어그램·폰트 변경 금지**. 헤더 #1F4E79, 서브헤더 #2E75B6, 강조 #D9E2F3, 회색 박스 #EEEEEE, 맑은 고딕, A4 1인치 여백.
5. **v2.x 누적 결정사항 변경 금지**. PLIP 보조지표, ADCP 부적합, MM-GBSA 중장기, 6개 패러다임, 13.4 절 제거는 모두 변경 불가.

### 새 세션 작업 시작 시 권장 순서

```
1. cd /sessions/.../outputs/build/
2. ls generate_v*.js → 가장 큰 번호 확인 (현재 v7)
3. cp generate_v7.js generate_v8.js
4. TZ='Asia/Seoul' date '+%Y%m%d_%H%M' → 새 파일명용 시각 확인
5. Edit 도구로 변경할 부분만 수정 (Read 후 Edit)
6. sed로 출력 파일명 변경 (이전 시각 → 새 시각)
7. node generate_v8.js 실행
8. validate.py 검증
9. 보고서 폴더로 cp
```

### 6-2. 헬퍼 함수 (generate_v*.js 공통) — 반드시 이것만 사용

| 함수 | 용도 | 호출 예시 |
|------|------|---------|
| `h1(text)` | 1장 헤딩 (1F4E79, 32pt, page break) | `children.push(h1("1. 문서 개요"))` |
| `h2(text)` | 2장 헤딩 (2E75B6, 26pt) | `children.push(h2("1.1 작성 목적"))` |
| `h3(text)` | 3장 헤딩 (404040, 22pt) | `children.push(h3("D-2. 학습 절차"))` |
| `p(text, opts)` | 일반 단락 (justified, 320 line spacing) | `children.push(p("본 문서는..."))` |
| `bullet(text, level)` | 한국어 불릿 `•` (level 0), `◦` (level 1) | `children.push(bullet([r("Label: ", { bold: true }), r("내용")]))` |
| `easyBox(title, lines[])` | 💡 회색 박스 (#EEEEEE, 회색 보더) | `children.push(easyBox("제목", ["line1", "line2"]))` |
| `quoteBox(text, source)` | 좌측 파란 보더 인용 박스 | `children.push(quoteBox("수식 또는 인용", "출처"))` |
| `makeTable(headers, rows, columnWidths, opts)` | 표 (헤더 #1F4E79 배경, 흰 글씨, 짝수 행 #F2F2F2 stripe) | `makeTable(["H1","H2"], [["A","B"]], [4000, 5026])` |
| `modelSubHead(letter, title)` | 모델 챕터 A/B/C/D/E/F 서브헤딩 | `children.push(modelSubHead("A", "한눈에 보기"))` |
| `blank(size)` | 빈 줄 (간격) | `children.push(blank())` |
| `r(text, opts)` | TextRun 생성 (bold, italic, color, size 등) | `r("강조 텍스트", { bold: true, color: "1F4E79" })` |

### 6-3. 표 셀의 컬러·강조 옵션

```javascript
// 강조 배경 (accentFill = #D9E2F3)
{ text: "내용", fill: accentFill, bold: true }

// 빨간색 글자 (주의/제거 표시)
{ text: "제외 결정", color: "C00000" }

// 회색 글자 (취소선 대신)
```

### 6-5. 알려진 함정

| 함정 | 대응 |
|------|------|
| Write 도구로 1,000줄+ 파일 작성 시 끝부분 잘림 (NULL 바이트) | `wc -l`로 끝까지 정상인지 확인. 잘림 발생 시 `head -n N` + 잘린 부분 별도 Write + `cat`으로 합치기 |
| PDF 미리보기에서 한글 깨짐 | LibreOffice 한글 폰트 부재 문제. docx 자체는 정상. `pandoc -t plain`으로 확인 가능 |
| `npm install -g docx` 권한 오류 | `cd outputs/build && npm install docx` (로컬 설치) |
| **sed -i 일괄 치환의 부작용 (v2.5 표지 오류 사례)** | **여러 위치에 같은 문자열이 있으면 의도하지 않은 곳도 함께 변경됨. 예: 출력 파일명만 바꾸려고 `sed -i 's/0034.docx/0213.docx/'` 실행 시, 표지의 "이전 버전: ...0034.docx (v2.3)"도 0213으로 바뀜. 반드시 `grep "치환대상" 파일`로 영향 범위 사전 확인. 가능하면 Edit 도구로 정확한 1곳만 변경할 것** |
| 한글 파일명 sed 치환 시 안전 | `\.docx` 정도만 escape, 한글은 그대로 가능 |

### 6-6. 자가 검토 체크리스트 (docx 생성 후 반드시 실행)

docx 생성 직후 다음 절차로 사실 정확성을 자가 검증해야 한다. validate.py만으로는 XML 구조만 확인되며 텍스트의 사실 오류는 잡히지 않는다.

```bash
# 1. XML 구조 검증
python3 ".claude/skills/docx/scripts/office/validate.py" output.docx

# 2. 표지 및 메타데이터 텍스트 추출 검증
pandoc output.docx -t plain 2>&1 | head -25

# 3. 변경 사항 표(1.2) 추출 검증
pandoc output.docx -t plain 2>&1 | sed -n '/1\.2/,/1\.3/p'

# 4. 참고문헌 추출 검증
pandoc output.docx -t plain 2>&1 | grep -E "\[1\]|\[14\]|\[26\]"
```

| 자가 검토 필수 항목 | 확인 내용 |
|------------------|---------|
| 표지 작성일 | KST 기준 새 시각인가 (이전 버전 시각 아님) |
| 표지 버전 번호 | 새 버전 번호로 갱신되었는가 |
| 표지 "이전 버전" 줄 | 이전 버전 파일명이 정확한가 (이번 새 버전 파일명과 혼동 없음) |
| 1.2 변경 사항 표 | 이전 버전 칼럼·이번 버전 칼럼 시각·내용 일치하는가 |
| 출력 파일명 (writeFileSync) | 새 시각인가 |
| 내부 본문 참조 | 변경 사항이 본문 절(D-2 등)에 실제 반영되었는가 |
| 13.2 교수님 안건 표 | 결정사항 표기 일관성 (MM-GBSA 중장기, ADCP 제외 등) |

위 4곳(표지, 1.2 표, 본문, 13.2 표)을 매번 pandoc으로 확인하면 v2.5 같은 표기 오류는 발생하지 않는다.
