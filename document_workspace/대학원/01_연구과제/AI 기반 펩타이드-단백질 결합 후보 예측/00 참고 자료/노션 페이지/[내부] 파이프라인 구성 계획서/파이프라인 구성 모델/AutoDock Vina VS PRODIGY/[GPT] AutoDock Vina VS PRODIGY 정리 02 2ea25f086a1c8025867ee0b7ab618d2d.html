[GPT] AutoDock Vina VS PRODIGY 정리 02

1. 두 모델의 본질적 차이

  • AutoDock Vina (스코어링만 사용)
    • 무엇을 하냐: 주어진 복합체 포즈(구조)에 대해 Vina의 경험적 도킹 스코어를 계산해서 “이 포즈가 얼마나 그럴듯한 결합 상태인지”를 빠르게 숫자로 요약
    • 모델 성격: 물리 기반이 아니라 경험적 스코어링 함수(빠른 근사)
    • 입력 관점: 수용체/리간드 준비(PDBQT)가 매우 중요(전하, 원자 타입, 토션 등 전처리 의존성 큼)
    • 출력의 의미: 절대 친화도(ΔG 정답)라기보다 포즈/후보 간 상대 비교 점수로 쓰는 게 안전
  • PRODIGY
    • 무엇을 하냐: 복합체 구조에서 인터페이스 접촉(contact)과 표면 특성(NIS 등)을 계산해 ΔG, Kd를 추정
    • 모델 성격: 구조 기반 특성(접촉 수/성질)로 만든 회귀식(접촉 기반 affinity predictor)
    • 입력 관점: 체인-체인 인터페이스가 명확해야 하고, 실제 접촉이 있어야 값이 의미 있음
    • 출력의 의미: ΔG/Kd 형태라 직관적이지만, 역시 정밀 절대값보다는 “구조 기반 상대 랭킹 신호”로 쓰는 게 안전(특히 펩타이드처럼 유연한 경우)

2. 우리 파이프라인에서 역할 비교

2-1. “무엇을 평가하나”가 다르다

  • Vina 스코어(포즈 탐색 제외)
    • 평가 대상: 리간드-수용체 상호작용을 Vina 스코어링 함수 관점에서 한 번에 요약
    • 강점: 빠름, 계산이 가벼움, 포즈/후보 대량 처리에 유리
    • 약점: PDBQT 전처리/부분전하/원자타입/리간드 취급 방식에 민감. 펩타이드는 원래 주 대상(소분자)보다 자유도가 커서 점수 해석을 보수적으로 해야 함
  • PRODIGY
    • 평가 대상: 인터페이스의 “접촉 네트워크(얼마나 붙어 있고, 어떤 성질의 접촉인지)”에 초점
    • 강점: 체인-체인 복합체 평가에 자연스러움(펩타이드가 별도 체인인 구조에 잘 맞음). ΔG/Kd로 리포팅이 쉬움
    • 약점: 접촉 기반이라 미세한 화학적 상호작용(특정 방향성 H-bond의 질, 전하 분포 세부 등)을 직접 모델링하는 게 아니라, 접촉 패턴으로 근사함. 포즈가 약간만 틀어져 접촉이 깨지면 급격히 나빠질 수 있음

2-2. “민감한 실패 모드”가 다르다

  • Vina는 이런 것에 민감
    • 전하/수소/원자타입 세팅 오류
    • 펩타이드 토션/유연성 처리의 한계
    • 포즈가 그럴듯해 보여도 스코어가 과대/과소될 수 있음(특히 절대값)
  • PRODIGY는 이런 것에 민감
    • 체인 지정 오류(체인 선택이 틀리면 의미 없음)
    • 인터페이스가 실제로 붙어 있지 않거나, 접촉이 적으면 예측이 급격히 나빠짐
    • 접촉은 많지만 “나쁜 접촉(클래시성, 비현실적)”도 접촉으로는 집계될 수 있으므로, 구조 튜닝/클래시 제거와 함께 써야 안정적

3. 왜 둘 다 써야 하나 (핵심 당위성)

  • 3-1. 서로 다른 관점의 독립 신호 제공 → 거짓양성(False Positive) 감소
    • Vina 스코어가 좋다 = “Vina 관점에서 상호작용이 그럴듯”
    • PRODIGY가 좋다 = “인터페이스 접촉 네트워크가 잘 형성”
    • 둘이 동시에 좋으면, 단일 지표만 좋을 때보다 “진짜로 잘 붙는 후보일 가능성”이 올라감
  • 3-2. 불일치 시 문제 유형 진단 가능 (파이프라인 운영상 매우 유용)
    • Vina 좋음 + PRODIGY 나쁨
      • 접촉이 실제로 적거나(포즈가 떨어짐), 체인/인터페이스 정의가 틀렸을 가능성
      • 또는 펩타이드가 ‘소분자처럼’ 스코어링은 그럴듯해도 체인-체인 접촉 패턴이 약한 포즈일 가능성
    • Vina 나쁨 + PRODIGY 좋음
      • 접촉은 많지만 Vina가 불리하게 보는 기하/전하 조건일 수 있음
      • PDBQT 전처리(전하/타입) 이슈 가능성
      • 또는 접촉은 촘촘하지만 화학적으로 질이 낮은(방향성/거리) 포즈일 수 있어 PLIP/GBSA로 추가 확인 가치가 큼
    • 둘 다 나쁨
      • 빠르게 탈락 후보로 분류 가능(계산 자원 절약)
  • 3-3. 계산 비용 계층화(멀티 스테이지 필터링) 구성에 유리
    • (가벼움) Vina 스코어링, PRODIGY
    • (중간) PLIP 상호작용 기반 필터
    • (무거움) OpenMM 기반 MM-GBSA 또는 추가 ML affinity
    • 단계별로 후보 수를 줄여 전체 처리량을 확보할 수 있음

4. 우리 파이프라인에서의 의미 (한 문장 요약)

  • AutoDock Vina (스코어만)
    • 빠르고 일관된 경험적 상호작용 점수로, 후보/포즈를 대량으로 1차 정렬하는 신호
  • PRODIGY
    • 인터페이스 접촉 네트워크 기반의 구조 친화도 추정(ΔG/Kd)으로, 체인-체인 결합의 구조적 타당성을 보강하는 신호

5. 추천 운용 방식 (파이프라인 규칙으로 연결)

  • Vina_score는 “빠른 1차 순위”로만 사용(절대 임계값에 과도하게 의존하지 않기)
  • PRODIGY ΔG/Kd는 “체인-체인 인터페이스가 제대로 형성됐는지”의 보조 근거로 사용
  • 최종 선별은 다중 근거로 묶는 방식이 안전
    • Vina 상위권 AND PRODIGY 상위권
      • + PLIP 상호작용 근거(핫스팟 잔기 포함, H-bond/salt bridge 등)
      • + (가능하면) GBSA/추가 ML affinity
