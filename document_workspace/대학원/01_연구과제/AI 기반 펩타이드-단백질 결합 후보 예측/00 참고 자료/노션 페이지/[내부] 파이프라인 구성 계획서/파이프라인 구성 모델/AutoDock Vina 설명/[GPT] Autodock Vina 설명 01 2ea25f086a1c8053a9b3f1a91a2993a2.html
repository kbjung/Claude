[GPT] Autodock Vina 설명 01

  • Vina = 구조 탐색 + 빠른 안정성 점수
  • 후보군 간 상대적 비교용.

구조 탐색(포즈 탐색)

  • Vina의 핵심은 “정답 포즈를 맞춘다”라기보다, 가능한 결합 자세(포즈)를 많이 만들어 보고 그중 그럴듯한 것들을 점수로 정렬하는 과정임.
  • 무엇을 탐색하나?
    • 위치/방향(translation/rotation): 리간드(펩타이드)가 단백질 주변에서 어디에, 어떤 방향으로 붙는지
    • 리간드 유연성: 리간드의 회전 가능한 결합(토션, dihedral)을 바꿔가며 형태(conformation)를 바꿈
    • (기본적으로) 단백질은 거의 고정: 대부분의 경우 리셉터는 rigid로 두고, 리간드만 유연하게 탐색함(리셉터 유연성은 제한적으로만 다룸)
  • 탐색 범위는 “박스(grid box)”로 정의함
    • conf.txt의 center_x/y/z, size_x/y/z로 “어디를 탐색할지”를 지정
    • 박스가 너무 작으면 결합부위를 놓치고, 너무 크면 탐색이 희석되어 성능이 떨어질 수 있음(속도도 느려짐)
  • 탐색 강도(얼마나 빡세게 뒤지나?)를 조절함
    • exhaustiveness: 탐색을 얼마나 촘촘히/많이 할지(올릴수록 더 오래 돌지만 더 다양한 포즈를 찾을 확률↑)
    • num_modes: 최종으로 출력할 상위 포즈 개수(“랭킹 후보 포즈”를 몇 개 남길지)
    • energy_range: 출력 포즈를 몇 kcal/mol 범위까지 포함할지(상위 포즈 주변의 대안 포즈 포함)
  • 최종 출력은 “여러 포즈 + 각 포즈 점수”
    • Vina는 단일 결과가 아니라, 상위 포즈들을 여러 개 내고(예: 9개) 각 포즈의 결합 에너지(점수)를 함께 제공함
    • 흔한 연구 흐름: 상위 포즈 선택 → PLIP/MM-GBSA/MD 등 후속 검증

간략 설명

  • 분자 도킹(Molecular Docking) 시뮬레이션 툴.
  • 입력(리셉터-리간드 3D 구조, 탐색 박스)을 바탕으로 결합 가능한 포즈를 탐색하고, 각 포즈를 scoring function으로 빠르게 평가해 결합 에너지(점수)로 랭킹함.
  • 단일 “정답”을 주기보다, 상위 포즈 여러 개를 제공해 후속 분석(상호작용/재스코어링/MD)으로 이어지기 좋음.

파이프라인 적합성

  • ColabFold 예측 구조의 물리적 타당성을 검증하는 '현실 체크' 단계로 활용.
  • AI 예측 구조에 대해 물리 법칙 기반의 에너지 값을 계산.
  • 구조 탐색(포즈 샘플링)
    • 같은 후보라도 “붙는 자세가 여러 개” 존재
    • “점수가 비슷한 대안 포즈들”을 확보해서 PLIP/PRODIGY 같은 후속 단계에서 더 믿을만한 포즈를 고를 수 있음
  • 이를 통해 전체 예측 결과의 신뢰도를 향상.

신뢰도

  • 신뢰도: 매우 높음 (Gold-Standard) 🏆
  • 근거
    • 분자 도킹 분야에서 가장 널리 알려진 표준 도구 중 하나.
    • 원본 논문이 3만 회 이상 인용될 정도로 학술적 신뢰도 확보.
    • 빠른 속도와 높은 정확도로 가상 스크리닝 연구에 널리 사용.

1. AutoDock Vina의 본질 한 줄 요약

AutoDock Vina는 수용체–리간드 복합체의 결합 자세(binding pose)를 탐색하고, 경험적 scoring function으로 ‘상대적인 결합 자유에너지(ΔG 유사값)’를 평가하는 docking 엔진입니다.

중요한 점은, 이 ΔG는 절대적인 열역학 ΔG가 아니라 docking score라는 것입니다.

2. Vina가 사용하는 에너지 모델 (Scoring Function)

Vina의 점수는 다음 항들의 **가중합(weighted sum)**입니다.

(1) Steric / van der Waals 항

  • 원자 간 거리 기반
  • 너무 가까우면 큰 패널티, 적당하면 favorable
  • Lennard–Jones 계열을 단순화한 형태

(2) Hydrophobic interaction

  • 비극성 원자 간 근접을 보상
  • 실제 소수성 효과(엔트로피 포함)를 경험적으로 근사

(3) Hydrogen bonding

  • donor–acceptor 거리 + 각도 조건
  • 방향성(directionality)을 반영

(4) Electrostatics (약하게 포함)

  • 정전기 항은 명시적으로 강하게 계산하지 않음
  • 대신 H-bond 항과 거리 기반 항에 간접적으로 흡수됨

(5) Conformational entropy penalty

  • 리간드의 rotatable bond 수에 비례한 패널티
  • “너무 유연한 분자는 결합 시 손해”라는 경험적 가정

(6) Overall score 형태

ΔGvina=w1⋅gauss1+w2⋅gauss2+w3⋅repulsion+w4⋅hydrophobic+w5⋅Hbond+w6⋅Nrot\Delta G_{\text{vina}} =
w_1 \cdot \text{gauss}_1
+ w_2 \cdot \text{gauss}_2
+ w_3 \cdot \text{repulsion}
+ w_4 \cdot \text{hydrophobic}
+ w_5 \cdot \text{Hbond}
+ w_6 \cdot N_{\text{rot}}

ΔGvina=w1⋅gauss1+w2⋅gauss2+w3⋅repulsion+w4⋅hydrophobic+w5⋅Hbond+w6⋅Nrot

→ 전부 실험 데이터로 fitting된 계수입니다.

3. Pose 탐색 방식 (Search Algorithm)

(1) Search space

  • 사용자가 정의한 grid box
  • 이 안에서만 docking 수행
  • box 설정이 결과 품질의 50% 이상을 좌우

(2) 탐색 알고리즘

  • Iterated local search + stochastic global search
  • Monte Carlo + gradient-based local optimization 혼합

(3) Output

  • 여러 개의 pose (ranked)
  • 각 pose마다 ΔG_vina (kcal/mol 단위)
  • RMSD 기준으로 clustering

4. Vina 점수의 해석 방법 (매우 중요)

절대값 해석 ❌

  • Vina에서 –8.0 kcal/mol이라고 해서

    “Kd가 몇 nM”이라고 말하면 틀림

상대 비교 ✔

  • 같은 타깃
  • 같은 binding site
  • 같은 grid
  • 같은 세팅

→ 이 조건에서만 ranking 도구로 의미 있음

1. 질문에 대한 결론부터 말씀드리면

네, AutoDock Vina의 scoring은 본질적으로

(1) 원자 간 거리 함수 × (2) 상호작용 타입별 가중치

의 조합입니다.

그리고 “의미 없을 정도로 멀리 있는 원자들”은 계산 대상에서 사실상 제외됩니다.

2. 1번: 원자 간 거리 기반 계산 — 어떻게 쓰이나?

Vina는 모든 원자쌍을 동일하게 다루지 않습니다.

(1) Pairwise atomic interaction 모델

  • 수용체 원자 i
  • 리간드 원자 j

    에 대해, 다음 형태의 항들을 계산합니다.

Eij(rij)=∑kwk⋅fk(rij)E_{ij}(r_{ij}) = \sum_k w_k \cdot f_k(r_{ij})

Eij(rij)=k∑wk⋅fk(rij)

  • rijr_{ij}rij: 원자 간 거리
  • fk(r)f_k(r)fk(r): 거리 의존 함수 (Gaussian, repulsion 등)
  • wkw_kwk: interaction type별 가중치

(2) 거리 함수의 핵심 특징

  • 짧은 거리: 강한 repulsion
  • 중간 거리 (접촉 범위): attractive
  • 멀어질수록: 빠르게 0으로 수렴

즉,

일정 거리 이상 떨어지면 계산은 수학적으로는 존재하지만, 값은 거의 0

3. 2번: 결합(상호작용) 종류에 따른 가중치 — 어떻게 반영되나?

Vina는 “이게 수소결합이다 / 이온결합이다”를 명시적으로 선언하지 않습니다.

대신 원자 타입(atom type) 조합으로 간접 반영합니다.

예시

  • O (acceptor) – H (donor)

    → H-bond 항 활성화

  • C (apolar) – C (apolar)

    → hydrophobic 항 기여

  • 원자 반경 + 타입

    → steric repulsion 판단

즉,

“결합 종류를 먼저 정하고 계산”이 아니라

“원자 타입 + 거리 → 자동으로 해당 항이 켜지거나 꺼짐”

4. 그럼 정말 모든 원자쌍을 계산하나?

이론적으로는 “가능”

실제 구현에서는 “아니다”

(1) Distance cutoff 존재

  • 일반적으로 8–10 Å 이상 떨어진 원자쌍:
    • 모든 fk(r)f_k(r)fk(r) ≈ 0
    • 계산에서 제외하거나 무시 가능

(2) Grid-based energy precomputation

  • 수용체는 미리 grid로 에너지 필드 계산
  • 리간드 원자는 grid에서 값 lookup

→ N² 원자쌍 직접 계산을 피함

(3) Docking box 밖 원자는 아예 고려 안 됨

  • box 바깥은 “존재하지 않는 공간”
  • 계산 대상 아님

5. 직관적 비유 (중요)

  • Vina는

    ❌ “모든 원자쌍을 끝까지 계산해서 합산”

    ✔ “가까워질 가능성이 있는 원자쌍만 에너지적으로 평가”

멀리 있는 원자:

  • 물리적으로도 상호작용 거의 없음
  • 수치적으로도 0
  • 연산적으로도 skip

결론 한 줄

AutoDock Vina에서는 docking box(= binding surface grid)를 벗어나는 원자·좌표는 에너지 계산 대상에서 아예 제외됩니다.

이는 “가중치가 작아진다”의 문제가 아니라, 계산 세계에 존재하지 않는다에 가깝습니다.

1. Docking box = 계산 우주의 경계

Vina에서 사용자가 정의하는 grid box는 단순한 탐색 범위가 아닙니다.

  • ✔ 에너지 필드가 정의된 공간
  • ✔ 리간드 원자가 존재할 수 있는 좌표계
  • ✔ scoring function이 유효한 영역

즉,

grid box 밖은 “에너지 함수가 정의되지 않은 영역”

입니다.

2. 실제 계산 구조에서 무슨 일이 일어나는가

(1) 수용체 전처리 단계

  • 수용체는 고정
  • grid box 내부에 대해서만:
    • 각 grid point마다
    • 원자 타입별 interaction energy를 미리 계산

→ box 밖에는 grid가 존재하지 않음

(2) docking 중 리간드 평가

  • 리간드의 각 원자 위치를:
    • grid 안에 투영 → 에너지 lookup
    • grid 밖 → 값 없음 → pose 자체가 invalid

보통은:

  • 강한 패널티
  • 또는 탐색 단계에서 즉시 reject

3. “일부 원자만 box 밖이면?”의 처리

이 부분이 실무에서 자주 헷갈리는 지점입니다.

경우 1. 리간드 전체가 box 밖

  • 탐색 대상 아님
  • docking 시작도 안 됨

경우 2. 리간드 일부 원자만 box 밖

  • 해당 pose는:
    • ✔ 에너지 계산 불완전
    • ✔ 물리적으로 “결합 실패 pose”

→ 탐색 알고리즘이 자동으로:

  • 해당 방향을 버리거나
  • 다시 box 안으로 끌어당김

즉,

“일부라도 벗어나면 그 pose는 살아남지 못함”

4. 왜 이렇게까지 강하게 자르는가?

이유 1. 계산 효율

  • docking은 수십만~수백만 pose 평가
  • grid 밖까지 고려하면 속도 붕괴

이유 2. 물리적 가정

  • “결합은 특정 binding surface에서 일어난다”
  • docking box = 실험적/구조적 사전 지식

이유 3. 탐색 안정성

  • box가 없으면 리간드가 공간을 떠돌며
    • meaningless local minima 발생

5. 이 개념을 정확히 이해하면 생기는 중요한 통찰

(1) box 설정이 결과의 절반이다

  • box를 너무 작게 잡으면:
    • 진짜 binding pose가 아예 평가되지 않음
  • 너무 크게 잡으면:
    • noise pose 증가
    • score 분산 증가

(2) “Vina score가 낮은데 말이 안 된다”의 흔한 원인

  • 진짜 상호작용 잔기가 box 경계 밖
  • 펩타이드 일부가 box를 벗어남
  • binding pocket이 아닌 표면 일부만 포함

6. 한 문장으로 최종 요약

AutoDock Vina에서는 binding surface로 정의된 grid box 내부만이 계산 세계이며, 이를 벗어나는 원자나 pose는 에너지 계산 대상에서 제외되어 docking 결과에 반영되지 않습니다.
